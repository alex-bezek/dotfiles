## Overview

This branch establishes a complete CI/CD pipeline and agent instruction framework for the dotfiles repository. It introduces:

1. **GitHub Actions CI workflow** — automated linting on PRs and pushes
2. **Comprehensive shell script linting** — shellcheck + bash syntax validation + JSON checks
3. **Agent instructions** — codified standards for both Claude agents and human contributors
4. **Install diagnostics** — portable troubleshooting script that auto-runs on install failure
5. **Status reporting** — exit code tracking for remote debugging (Codespaces, CI, etc.)
6. **ShellCheck configuration** — `.shellcheckrc` for suppressing false positives

- **`.github/copilot-instructions.md`** — Calibration guide for Copilot/Claude agents
  - Explicitly states what IS and IS NOT a code review issue in this repo
  - Default stance: approve unless critical issue detected
  - Defines review intensity for different PR types
  - Clarifies that "non-standard" style is not a bug

#### `scripts/ci/`
- **`scripts/ci/lint.sh`** — Deterministic linting script (138 lines)
  - **File discovery logic:**
    - All `.sh` files anywhere in the repo (except `.git/`)
    - Extensionless files in `./git/hooks/` and `./claude/hooks/`
    - Scripts in `./scripts/` with bash shebang
    - Deduplicates and excludes `.zshrc` and `p10k.zsh`
  - **Three checks:**
    1. ShellCheck (per file) — captures portability and style issues
    2. Bash syntax (`bash -n`) — catches parse errors
    3. JSON validation (`jq empty`) — validates all `.json` files
  - **Graceful degradation:** if a tool is missing (e.g., jq in some CI environments), it warns and skips
  - **Exit code:** 0 if all pass, 1 if any fail
  - **Output:** colored, human-readable (✅/❌/⚠️ emoji)

- **`scripts/dotfiles-debug.sh`** — Portable diagnostics (124 lines)
  - Auto-triggered by install.sh trap on failure
  - Can also run standalone: `bash scripts/dotfiles-debug.sh`
  - Checks:
    - OS info (uname, `/etc/os-release`, `$OSTYPE`)
    - Tool availability (zsh, bash, git, brew/apt, etc.)
    - Shell rc file paths (`.zshrc`, `.bashrc`, `.profile`)
    - Symlink validity (broken vs missing vs regular files)
    - Critical directories (claude, themes, etc.)
  - Works on macOS and Linux (GNU vs BSD tool detection)
  - Safe to run in restricted environments (pipes, CI, containers)

#### `.shellcheckrc`
```
disable=SC1091  # Can't follow non-constant source (e.g., vendor scripts)
disable=SC2030  # local OS in subshell doesn't affect outer scope
disable=SC2031  # global OS not affected by subshell assignment
```
These allow the install.sh script to use `local OS` in a subshell without false positives.

#### `AGENTS.md`
Formal agent instructions defining:
- **Git workflow:** feature branches, conventional commits, squash merge, PR gate
- **Shell script conventions:** `set -euo pipefail`, `$HOME` not hardcoded, shellcheck + bash -n compliance
- **File permissions:** all new `.sh` and hooks must be executable
- **Style:** match neighboring files, no refactoring beyond scope
- **Protected files:** don't touch `p10k.zsh`, `/Applications/`, `/System/`, `/Library/`
- **Key directories:** table of what each major dir manages (claude/, themes/, scripts/, etc.)

## Context & Conversation History

### Problem Being Solved

**Issue:** The dotfiles repo has grown to support multiple platforms (macOS ARM64, macOS Intel, Linux, GitHub Codespaces, devcontainers) and environments (remote SSH, containers, CI). Without a centralized linting gate, script changes risked:
- Portability regressions (GNU vs BSD flags, missing tool guards)
- Shell syntax errors only caught at runtime
- Inconsistent style across hooks and scripts

**Goal:** Establish a single source of truth for code review standards and automate the linting gate.

### Why This Approach

1. **Linting script, not pre-commit hook:** The repo explicitly avoids setup.py/node_modules/ruby gem dependencies. `lint.sh` uses only standard tools (bash, find, shellcheck, jq) available on GitHub Actions and locally.

2. **GitHub Actions, not pre-commit.com:** Avoids external SaaS dependency; full control and offline capability.

3. **Diagnostics script as safety net:** Install can fail silently in headless environments (CI, containers, Codespaces). Auto-running diagnostics on exit provides visibility without manual debugging.

4. **Agent instructions explicit:** Many agents make assumptions about code style. This repo is deliberately "non-standard" in places (no type annotations in shell, no docstrings). Instructions prevent agents from flagging non-issues as review blockers.

## Testing & Verification Plan

### Local Testing

**Before merging, verify:**

1. **Lint script runs without errors on this branch:**
   ```bash
   bash scripts/ci/lint.sh
   ```
   Expected output:
   ```
   🔍 Found N shell files to check
   ━━━ ShellCheck ━━━
   ✅ ShellCheck passed (N files)
   ━━━ Bash Syntax ━━━
   ✅ Bash syntax passed (N files)
   ━━━ JSON Validation ━━━
   ✅ JSON validation passed (M files)
   ✅ All checks passed
   ```

2. **Install script runs successfully (or expected failure is captured):**
   ```bash
   DOTFILES_DIR=. bash install.sh 2>&1 | tail -20
   cat ~/.dotfiles-status
   ```
   Should show `status: success` with timestamp and platform.

3. **Diagnostics script runs without errors:**
   ```bash
   bash scripts/dotfiles-debug.sh
   ```
   Should produce readable output for all sections.

### CI Testing

**GitHub Actions workflow will:**
1. Checkout the PR branch
2. Install shellcheck + jq on Ubuntu
3. Run `bash scripts/ci/lint.sh`
4. Pass if exit code is 0; fail if exit code is 1

**Note:** The workflow triggers on:
- Pull requests (any branch → master)
- Pushes to master

### Cross-Platform Testing
